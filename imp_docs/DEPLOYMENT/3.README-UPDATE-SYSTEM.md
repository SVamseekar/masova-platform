## Overview

### The Challenge Solved

**Question:** After deploying MaSoVa as installable software, how do you push bug fixes to customers?

**Answer:** Automated update system with Docker Hub (FREE) and GitHub Actions (FREE)

### Three Development Approaches

| Approach | Description | When to Use |
|----------|-------------|-------------|
| **Microservices Dev** (Recommended) | Develop in microservices, build monolith for deployment | Fast iteration, clean separation |
| **Direct Monolith Dev** | Develop directly in monolith | Simpler for small teams |
| **Hybrid** | Dev in microservices, test in monolith | Best of both worlds |

**This system supports ALL THREE approaches!**

## Development Workflow

### Option 1: Microservices Development (Recommended) ⭐

**Workflow:**
```
1. Fix bug in microservice
   ├── cd order-service
   ├── vim src/main/java/.../OrderService.java
   ├── git commit -m "Fix: Order status bug"
   └── git push

2. Build monolith (automated or manual)
   └── ./build-monolith.sh --docker --version=1.0.2

3. Push to Docker Hub (automated via CI/CD)
   └── docker push masova/backend:1.0.2

4. Customer updates
   └── ./update-masova.sh
```

**Time from bug fix to customer: 10-15 minutes**

### Option 2: Direct Monolith Development

**Workflow:**
```
1. Fix bug in monolith code
   ├── cd masova-monolith
   ├── vim src/main/java/.../OrderService.java
   └── git commit

2. Build Docker image
   └── docker build -t masova/backend:1.0.2 .

3. Push
   └── docker push masova/backend:1.0.2

4. Customer updates
   └── ./update-masova.sh
```

### Option 3: Hybrid (Best for Teams)

**Workflow:**
```
Developer A: Works on order-service (microservice)
Developer B: Works on payment-service (microservice)

Build Team:
  └── ./build-monolith.sh  # Combines all microservices
  └── Test monolith
  └── Push to Docker Hub

Customers:
  └── Get unified monolith deployment
```

### Automatic Update Checking

**Built-in UpdateService:**
- Checks for updates daily at 2 AM
- Compares `version.properties` on GitHub (FREE - no API limits)
- Notifies admin if update available
- Optional: Auto-install patch versions (1.0.1 → 1.0.2)

**Version Types:**
```
Patch (1.0.1 → 1.0.2)    → Auto-update OK ✅
Minor (1.0.2 → 1.1.0)    → Manual update recommended
Major (1.1.0 → 2.0.0)    → Manual update required (breaking changes)
```

### Update Scripts

#### 1. `update-masova.sh` (Customer)

**What it does:**
```bash
./update-masova.sh

# Automatically:
✅ Creates backup of current version
✅ Backs up .env file
✅ Backs up database
✅ Downloads new version
✅ Updates Docker images
✅ Restarts services
✅ Verifies health
✅ Rolls back on failure
```

**Options:**
```bash
./update-masova.sh --version=1.0.2   # Update to specific version
./update-masova.sh --auto            # Skip confirmations
./update-masova.sh --skip-backup     # Skip backup (not recommended)
```

#### 2. `rollback-masova.sh` (If update fails)

```bash
./rollback-masova.sh

# Lists available backups
# Allows selecting version to restore
# Automatically restores database and .env
```

## Troubleshooting

### Build Issues

**Problem:** `build-monolith.sh` fails
```bash
# Check Maven
mvn --version

# Check Java
java -version  # Should be 17+

# Clean and rebuild
mvn clean
./build-monolith.sh
```

**Problem:** Docker build fails
```bash
# Check Docker
docker --version

# Login to Docker Hub
docker login

# Try building without cache
docker build --no-cache -t masova/backend:test .
```

### Update Issues

**Problem:** Update script can't find new version
```bash
# Check internet connection
curl https://raw.githubusercontent.com/yourusername/masova/main/version.properties

# Check Docker Hub
docker pull masova/backend:latest

# Update manually
docker-compose pull
docker-compose up -d
```

**Problem:** Services don't start after update
```bash
# Check logs
docker-compose logs

# Rollback
./rollback-masova.sh

# Check .env file
cat .env  # Make sure all required variables are set
```

### GitHub Actions Issues

**Problem:** CI/CD pipeline fails
```bash
# Check GitHub Actions tab
# Common issues:
# - Docker Hub credentials not set
# - Build timeout (increase in workflow)
# - Maven dependencies issue (add caching)
```

## Best Practices

### Version Numbering

Use semantic versioning: `MAJOR.MINOR.PATCH`

```
1.0.0 → 1.0.1    Patch: Bug fix (safe for auto-update)
1.0.1 → 1.1.0    Minor: New feature (notify user)
1.1.0 → 2.0.0    Major: Breaking change (require manual update)
```

### Changelog Maintenance

Always update `CHANGELOG.md`:

```markdown
## 1.0.2 (2025-12-25)

### Bug Fixes
- Fixed order status not updating in real-time (#123)
- Corrected delivery fee calculation for Zone 3 (#124)

### Improvements
- Improved kitchen display performance by 30%
```

```bash
# 1. Test in microservices
mvn test

# 2. Build monolith
./build-monolith.sh

# 3. Test monolith locally
cd masova-monolith

# 4. Integration test
# - Create order
# - Process payment
# - Update kitchen
# - Complete delivery

# 5. Tag and release
git tag v1.0.2
git push origin v1.0.2
```

### Backup Strategy

**Automated (in update script):**
- Database backup before every update
- `.env` file backup
- Previous version JARs kept

**Manual (for critical updates):**
```bash
# Full system backup
docker-compose exec mongodb mongodump --out /backup/manual-$(date +%Y%m%d)
cp -r ~/MaSoVa ~/MaSoVa-backup-$(date +%Y%m%d)
```

## FAQ

### Q: Do I need a separate monolith codebase?
**A:** No! The `build-monolith.sh` script creates the monolith from your microservices automatically. You develop in microservices, deploy as monolith.

### Q: Is Docker Hub really free?
**A:** Yes! Public images are free forever. Unlimited repositories, unlimited pulls.

### Q: What if a restaurant doesn't want Docker?
**A:** The system supports both Docker and standalone JARs. They can run JARs directly.

### Q: How long does an update take?
**A:** 2-3 minutes total. ~30 seconds downtime.

### Q: Can I rollback an update?
**A:** Yes! `./rollback-masova.sh` with automatic database restore.

### Q: What if GitHub Actions fails?
**A:** You can still build manually with `./build-monolith.sh` and push to Docker Hub.

### Q: Is auto-update safe?
**A:** Only for patch versions (bug fixes). Minor/major versions require manual confirmation.

### Q: What about data during updates?
**A:** Data is preserved. Docker volumes persist. Database is backed up automatically.
